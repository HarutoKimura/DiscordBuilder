# DiscordBuilder — multiplayer vibe coding

**Your Discord server becomes a development studio.** A community member types
`/build <what they want>` in plain language; OpenAI Codex implements a
full-stack app in a sandbox and posts a live URL back to the thread. Anyone
who wants a change just replies. Nothing ships until the community votes 👍.

## Why

Every community needs small tools — a shift table for the festival, an RSVP
page, a book-club vote. Nobody builds them: the people who feel the need
can't code, and the people who can code don't have the time. DiscordBuilder
moves software creation into the place the community already lives, and makes
the *conversation itself* the spec, the changelog, and the release process.

- **The thread is the spec.** The original request and every "fix this"
  reply form the complete, readable history of how the app came to be.
- **The reaction is the release gate.** 👍 votes from members (configurable
  threshold) approve a version; the bot announces the launch to the channel.
- **Data is sacred.** Edit builds preserve everything members have entered —
  the agent contract forbids destructive changes without an explicit,
  surfaced `dataReset` flag.

## How it works

```
Discord (discord.js v14 — slash command, threads, reactions)
   │  /build request … → thread → streamed progress → result embed
   ▼
Orchestrator (Node.js/TypeScript, single process)
   │  per-project build queue (serial per project, parallel across projects)
   │  summarizes the Codex event stream into ONE edited status message
   ▼
Sandbox (one Docker container per project)
   │  app template + codex CLI; `codex exec` edits ONLY inside the template
   ▼
Deploy target (per-project Cloudflare quick tunnel → public HTTPS URL)
```

1. `/build` copies the app template (Next.js App Router + SQLite via Drizzle)
   into a fresh Docker container and runs **`codex exec --json`**
   (model: GPT-5.6 Sol) against it.
2. The orchestrator parses the Codex event stream live and renders it into a
   single Discord status message: commands run, files changed, and what Codex
   says it is doing — waiting is part of the show, not dead air.
3. When Codex finishes it must write a **`BUILD_RESULT.json`** contract
   (status, plain-language summary, changes, screenshots, data-reset flag).
   The bot turns it into an embed with screenshots and an "Open the app"
   button.
4. **Any human reply in the thread becomes an edit build** on the same
   project, queued serially so the conversation stays coherent.
5. After every successful build the bot opens a 👍 vote. Enough votes →
   the version is approved, the vote closes, and a launch announcement goes
   to the parent channel.

## The Codex integration (the interesting part)

The runtime code generator is **OpenAI Codex** (`codex exec`, non-interactive,
model GPT-5.6 Sol). The integration is not a thin API call:

- **Bounded authority.** Codex works inside a per-project Docker container
  and is only allowed to edit application code. Auth, DB client setup, and
  deploy config are human-owned template files, enforced by
  [`AGENTS.md`](templates/app-template/AGENTS.md) — the instructions Codex
  reads at runtime.
- **A design system as a hard contract.**
  [`DESIGN.md`](templates/app-template/DESIGN.md) pins a monochrome,
  documentation-first visual system (tokens pre-wired into Tailwind). Every
  generated app looks like part of the same product family instead of
  default-AI output.
- **A self-check quality loop.** Codex must run typecheck, a production
  build, DB migrations, and a Playwright screenshot pass — desktop *and*
  390px mobile — and look at what it built before finishing. Cramped phone
  UI is a build failure, not a shrug.
- **Structured output, untrusted by default.** `BUILD_RESULT.json` is
  validated field-by-field. Screenshot paths are opened through an
  `O_NOFOLLOW` file descriptor with same-inode verification so a compromised
  build can't symlink host secrets into a Discord upload.
- **Event-stream UX.** The `--json` event stream drives the live progress
  message, and every build's full stream is archived under `var/projects/`
  as the project's development record.

## Getting started

Prerequisites: Docker Desktop, Node.js 22 + pnpm, `codex` CLI (logged in),
`cloudflared` (`brew install cloudflared`), and a Discord application with the
**Message Content** privileged intent enabled.

```sh
git clone https://github.com/HarutoKimura/DiscordBuilder && cd DiscordBuilder
pnpm install
cp .env.example .env    # fill in DISCORD_BOT_TOKEN / DISCORD_CLIENT_ID
docker build -t discordbuilder-sandbox infra/docker
pnpm dev                # starts the bot + orchestrator
```

Then, in any text channel of a server the bot has joined:

```
/build request: An RSVP page for our next meetup, with a headcount.
```

`pnpm cli build "<prompt>"` runs the same pipeline without Discord.
`SHIP_APPROVAL_VOTES` sets the 👍 threshold (default 2).
`DEPLOY_MODE=local|cloudflared` chooses private localhost URLs or public
quick-tunnel URLs.

## Trust & safety

- **Private by default.** `DEPLOY_MODE=local` serves apps only on the host;
  public tunnels are an explicit opt-in. (Production design: Caddy on a VPS
  with a Discord-OAuth gate — see roadmap.)
- **Everything the agent writes is treated as untrusted input** — validated
  result contract, symlink/TOCTOU-guarded uploads, bounded screenshot
  processing.
- **AI-reviewed CI.** Every PR to `main` triggers an adversarial code review
  ([workflow](.github/workflows/adversarial-review.yml)): two split-context
  reviewer agents receive only the diff, assume the code is wrong, and must
  back every finding with a concrete failure scenario. 30+ confirmed findings
  were caught and fixed this way during development.

## Roadmap

- Discord-OAuth gate at the proxy so only server members can open an app
- Stable per-app subdomains on a VPS (Caddy wildcard) instead of quick tunnels
- Faster edit turnarounds and richer app templates
